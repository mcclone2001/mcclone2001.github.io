The Twelve Factor App

March 1, 2019

“The twelve-factor app” es una metodología para construir aplicaciones SaaS propuesta por Adam Wiggins co-fundador de Heroku, a finales del 2011.

Esta publicación es un resumen de los 12 factores, y es el inicio de una serie de 13 artículos dirigidos a analizar cada uno de estos puntos.

Si bien ha recibido controversia, dado que los 12 factores se ajustan casi exclusivamente a los flujos de trabajo con Heroku, el manifiesto me ha parecido interesante dado que varios de los puntos están evidentemente inspirados por los movimientos de DevOps y la arquitectura de microservicios, pero principalmente por ser una síntesis de los 6 años de experiencia de Wiggins como CTO de una de las primeras plataformas en la nube.

1.- Código Base

Gestionar el código mediante sistemas de control de versiones (Git, Subversion, etc.)

El mismo código no debe repetirse en más de un repositorio, si es necesario reusar código debe hacerse separando dicho código en librerías (y por tanto en otro repositorio) que puedan enlazarse mediante gestores de dependencias (NPM, Composer, etc.)

El código base puede desplegarse múltiples veces, un despliegue es una instancia de la aplicación que está en ejecución (El servidor de producción, el servidor de pruebas, el equipo del desarrollador, son todos instancias de la aplicación), aunque cada instancia puede tener una versión diferente (diferentes commits de un mismo repositorio)

2.- Dependencias

Las aplicaciones no deben depender de que en el sistema donde se ejecuten existan librerías especificas instaladas (p.ej. mi programa necesita que instales direct x versión 10.0 para que pueda arrancar), ni deben depender de que exista paquetería específica instalada en el sistema (ImageMagick, 7zip, curl), dado que no existe la seguridad de que existan en todos los entornos donde se desplieguen.

Todas sus dependencias deben estar especificadas en un manifiesto de declaración de dependencias (cómo el Package.json de NPM, ó el Composer.json de Composer)

La aplicación también debe funcionar dentro de entornos aislados que no puedan afectar al sistema donde se ejecutan (cómo Virtualenv para Python)

3.- Configuraciones

La configuración de una aplicación es todo lo que puede variar entre despliegues, por tanto deben estar fuera del código base. La prueba de fuego sería poder compartir el código base sin riesgo de que se expongan usuarios o contraseñas de ningún tipo.

La recomendación es guardar la configuración en variables de entorno (Los viejos confiables comandos SET en windows y EXPORT en Linux)

4.- Backing services

Los “backing services” son los servicios que la aplicación consume (cómo bases de datos, almacenamiento de archivos, correos o cualquier otro servicio disponible a través de una API) estos servicios, sean locales o de terceros, deben poder cambiarse sin necesidad de hacer modificaciones al código base del sistema.

Es decir, las URLs y credenciales de estos recursos conectados no deben formar parte del código base, si tu aplicación hace uso de una base de datos MariaDb y decides migrarla a otro servidor para escalar tu aplicación verticalmente, lo único que deberías necesitar es modificar las variables de entorno que tu aplicación utiliza para almacenar la URL, nombre de usuario y contraseña.

5.- Construir, desplegar, ejecutar

El código base se transforma en despliegue después de haber sido construida, desplegada y estar ejecutándose.

Las aplicaciones 12 factor deben separa estas tres fases, un bug jamás debe resolverse en el código que se está ejecutando en el servidor, cada versión generada debe tener un identificador único y almacenarse en un sistema de archivado para, en caso de errores catastróficos, poder regresar a versiones anteriores con el menor impacto posible.

6.- Procesos

La aplicación se ejecuta como procesos sin estado, cualquier información que necesite debe almacenarse mediante backing services, los cuales sí pueden tener estados persistentes (como bases de datos, o sistemas de archivos), sin embargo, la aplicación puede almacenar información temporal directamente a sistemas de archivos, memoria, etc., siempre y cuando sea solo como caches temporales y no se espere que la información almacenada en ellos exista más allá de la ejecución de un proceso.

7.- Asignación de puertos

Las aplicaciones deben ser totalmente auto contenidas, es decir, no deben depender de un servidor web (cómo Apache, Nginx, IIS) para crear un servicio web público.

Las aplicaciones deben ligarse a un puerto y a través de él publicar sus servicios, esto permite no solo que los consuma un cliente, sino que a su vez pueden convertirse en ‘backing services’ para otras aplicaciones.

8.- Concurrencia

Las aplicaciones tienen la necesidad de escalarse en algún momento, escalar una aplicación verticalmente tiene un costo y límite, por tanto tratar a las aplicaciones como procesos permite que escalarlos horizontalmente sea fácil, únicamente es necesario crear nuevas instancias de la aplicación.

Por tanto, es importante dividir la aplicación en distintos tipos de procesos, por ejemplo las peticiones web pueden resolverse con procesos diseñados para ser rápidos en ejecución y delegar las tareas largas a otro tipo de procesos, tomamos la inspiración del modelo de procesos de Unix.

9.- Posibilidad de ser desechados

Los procesos de las aplicaciones deben ser desechables, es decir deben poder ser iniciados o terminados en cualquier momento.

Terminar un proceso significa dejar de recibir nuevas solicitudes y concluir las tareas de las solicitudes que ya estaban procesándose. Este enfoque facilita la escalabilidad del sistema y considerarlo, desde el diseño de la arquitectura, permite hacer una aplicación resistente a fallos, el diseño “crash only” es el mejor ejemplo de este concepto.

10.- Paridad entre ambientes de desarrollo y producción

Minimizar el tiempo que transcurre entre la codificación y la puesta en producción es vital para responder a los cambios del mercado, las aplicaciones deben desarrollarse en ambientes lo más parecidos posible a los ambientes en que se ejecutarán, para minimizar los errores implícitos en omitir esta práctica.

11.- Historiales

Los historiales deben verse como la transmisión de eventos ordenados y capturados de la salida de los procesos de la aplicación y de sus ‘backing services’.

Dichos eventos se envían a la salida estándar y los entornos de ejecución deberán encargarse de indizarlos, analizarlos y almacenarlos, esto jamás será responsabilidad de la aplicación.

12.- Procesos Administrativos

Frecuentemente los desarrolladores desean ejecutar procesos administrativos pensados en ser ejecutados una sola vez, o de manera extemporánea, por ejemplo: * Modificar estructuras de bases de datos * Corregir registros * Ejecutar diagnósticos

Las aplicaciones deben proveer de un servicio mediante consola para poder ejecutar estos procesos, todas las operaciones de mantenimiento deben existir en el código base de la aplicación.
