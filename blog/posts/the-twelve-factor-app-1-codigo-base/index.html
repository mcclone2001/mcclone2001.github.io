The Twelve Factor App

March 25, 2019

“The twelve-factor app” es una metodología para construir aplicaciones SaaS propuesta por Adam Wiggins co-fundador de Heroku, a finales del 2011.

Esta publicación es un breve análisis del primer punto.

Te recomiendo leer el resumen de los doce puntos antes de continuar con este artículo.

1.- Código Base

Gestionar el código mediante sistemas de control de versiones (Git, Subversion, etc.)

El mismo código no debe repetirse en más de un repositorio, si es necesario reusar código debe hacerse separando dicho código en librerías (y por tanto en otro repositorio) que puedan enlazarse mediante gestores de dependencias (NPM, Composer, etc.)

El código base puede desplegarse múltiples veces, un despliegue es una instancia de la aplicación que está en ejecución (El servidor de producción, el servidor de pruebas, el equipo del desarrollador, son todos instancias de la aplicación), aunque cada instancia puede tener una versión diferente (diferentes commits de un mismo repositorio)

Sistema de Control de Versiones

Para el que aún no conoce lo que es un sistema de control de versiones, wikipedia lo define como

Se llama control de versiones a la gestión de los diversos cambios que se realizan sobre los elementos de algún producto o una configuración del mismo. Una versión, revisión o edición de un producto, es el estado en el que se encuentra el mismo en un momento dado de su desarrollo o modificación.

Uno de los sistemas de control de versiones más conocidos es Git y es el utilizado en el mundialmente famoso GitHub (duh!).

Sobresimplificando sus funcionalidades podemos decir que lo que hace es llevar un historial incremental de cambios sobre los archivos de un proyecto, permitiendo a un equipo de desarrolladores trabajar sobre el mismo proyecto y revisar, mezclar y distribuir los cambios realizados a todo el equipo de una manera ordenada y, dependiendo de las prácticas del equipo, controlada.

Un repositorio es una carpeta, y todo su contenido, controlada por el sistema de control de versiones.

Es posible organizar el código de un proyecto en multiples repositorios, especialmente cuando se considera que cierto código puede llegar a utilizarse en mas de un proyecto.

Para ahondar en como explotar los sistemas de control de versiones, existen un par de flujos de trabajo orientados a git que son de gran interés, “Git workflow” y “Trunk based development”.

Gestores de Dependencias

Actualmente es común que en todos los proyectos se hace uso de librerias de terceros, buscando obtener la mayor velocidad de desarrollo posible se evaluan librerias, frameworks y utilidades, principalmente de código abierto, que proveen de funcionalidades cuyo costo de producción puede ser de moderado a alto, y por tanto preferimos no reinventar la rueda.

Es también común que al no tener el control sobre el desarrollo de estas librerías ocurran actualizaciones a estas, que inserten algún bug o modifiquen la API que proveen, y esto en consecuencia crea problemas difíciles de predecir en nuestro sistema.

Por tanto usualmente decidimos “congelar” las versiones de las librerías que utilizamos para asegurar estabilidad durante nuestro desarrollo y despliegue en producción.

Las herramientas de control de dependencias nos permiten gestionar facilmente que librerías y que versiones de las mismas se están utilizando.

Algunas de las herramientas de control de dependencias (también conocidos como “package managers”) más populares son NPM para Javascript, PIP para Python y Composer para PHP.

Mediante manifestos de dependencias es posible garantizar que un proyecto puede ser ejecutado o recompilado en distintos entornos sin la dificultad que implica administrar los árboles de dependencias que existen entre las muchas librerías del proyecto.

Despliegues

Un despliegue es una ejecución de tu código fuente.

Suponiendo que tienes un software X, cada equipo que esta ejecutando dicho software es un despliegue del mismo.

Mientras que todos los equipos están ejecutando la misma versión del software todos compartiran el mismo código base.

Cuando modifiques el software y la mita de los equipos esten ejecutando la versión anterior y la otra mitad la nueva, todos siguen siendo despliegues del mismo código base, pero de dos versiones distinas del mismo.

Una de las recomendaciones más frecuentes para minimizar problemas durante los despliegues es que los desarrolladores dispongan de un ambiente de trabajo tan cercano como sea posible a los ambientes de producción en los que el software se ejecutará.

Es una práctica común el uso de contenedores para proveer de estos ambientes preconfigurados y reducir la posibilidad de que un desarrollador trabaje en un ambiente distinto y caigamos en aquella famosa situación de “Profe en mi máquina si jalaba”.
